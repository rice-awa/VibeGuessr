# 图片模糊、等待提示与骨架屏优化方案

## 背景

当前 VibeGuessr 的核心玩法依赖“AI 生成模糊图片 + 玩家猜词”。用户反馈了三个体验问题：

1. 简单模式下图片看起来不模糊。
2. 等待图片生成时，提示文案固定为“先根据分类判断，图片马上补齐。”，不会轮播其他提示。
3. 图片区域等待状态偏纯白、单调，希望改成方形骨架屏。

本文档基于当前代码实际行为做根因分析，并给出可落地的优化方案。目标是避免凭猜测修改，先明确数据流、责任边界、变更范围和验证方式。

---

## 当前代码事实

### 1. 后端图片模糊逻辑

相关文件：

- `backend/config.py`
- `backend/services/image_service.py`
- `backend/services/game_service.py`
- `backend/app.py`

当前难度配置中只有 prompt 级别的模糊描述：

```python
# backend/config.py
DIFFICULTY_CONFIG = {
    "easy": {
        "blur_prompt": "slightly blurry, soft focus, the outline is recognizable",
        "image_strategy": "generate a complete image based on the visual description",
    },
    "medium": {
        "blur_prompt": "blurry, out of focus, abstract, partial close-up or unusual angle",
        "image_strategy": "generate a partial close-up or non-typical angle",
    },
    "hard": {
        "blur_prompt": "very blurry, heavily abstracted, minimal detail, almost unrecognizable",
        "image_strategy": "extract only abstract visual elements",
    },
}
```

图片生成服务只是把 `blur_prompt` 拼进文生图 prompt：

```python
# backend/services/image_service.py
def generate_image(visual_desc, blur_prompt, image_strategy):
    prompt = (
        f"{image_strategy}. "
        f"Visual description: {visual_desc}. "
        f"Style: {blur_prompt}. "
        f"Do NOT include any text, letters, or words in the image."
    )
    return _request_image(prompt)
```

结论：当前没有真正的图像后处理，也没有高斯模糊。所谓“模糊”完全依赖模型是否遵守 prompt。

### 2. 简单模式不模糊的根因

简单模式使用的是：

```text
slightly blurry, soft focus, the outline is recognizable
```

这条提示本身偏弱，且图片模型可能为了生成高质量结果而弱化或忽略“soft focus”。因此简单模式返回清晰图是当前设计的自然结果，不是前端展示 bug。

真正问题是：

- 模糊强度没有确定性保证。
- 后端没有保存“原始图”和“模糊图”的明确语义边界。
- 测试只验证图片响应能解析，不验证模糊处理是否执行。

### 3. 等待提示逻辑

相关文件：

- `frontend/src/hooks/useGame.js`
- `frontend/src/components/LoadingOverlay.jsx`
- `frontend/src/pages/Game.jsx`
- `frontend/src/pages/questionPresentation.js`

在 SSE 的 `word_ready` 阶段，前端硬编码 fallback 文案：

```javascript
// frontend/src/hooks/useGame.js
setFallbackHint('先根据分类判断，图片马上补齐。')
```

同样逻辑出现两处：

- 首题启动流：`startNewGame` 内的 `onWordReady`
- 下一题流式加载：`loadNextQuestion` 内的 `onWordReady`

当前没有等待文案数组、轮播索引或定时器。因此用户看到的等待提示固定不变。

### 4. 图片占位与骨架屏逻辑

相关文件：

- `frontend/src/pages/Game.jsx`
- `frontend/src/pages/Game.css`

当前图片加载中使用：

```jsx
<div className="gm-image-skeleton">
  <div className="gm-image-skeleton-pulse" />
</div>
```

样式为：

```css
.gm-image-skeleton {
  position: absolute;
  inset: 0;
  display: flex;
  align-items: center;
  justify-content: center;
  background: #f1f5f9;
}

.gm-image-skeleton-pulse {
  width: 60px;
  height: 60px;
  border-radius: 50%;
  background: var(--color-border);
  animation: pulse 1.5s ease-in-out infinite;
}
```

当前骨架屏只是浅灰底 + 圆形脉冲，没有体现“图片正在生成”的方形图片占位，也缺少视觉层次。

### 5. 流式出题数据流

当前首题和下一题都走 `/api/game/next/stream`：

```text
后端生成关键词 -> emit word_ready -> 前端进入 partial -> 后端生成图片 -> emit image_ready -> 前端进入 playing
```

关键路径：

- 后端 `backend/app.py` 的 `next_question_stream`
- 前端 `frontend/src/services/api.js` 的 `streamNextQuestion`
- 前端 `frontend/src/hooks/useGame.js` 的 `onWordReady` / `onImageReady`

这说明等待体验优化应该主要放在前端，不需要改 SSE 协议。

## 目标

### 必须解决

1. 生成图片后，后端对用于猜题的图片执行确定性的高斯模糊。
2. 简单、中等、困难模式有不同但可控的模糊半径。
3. 图片生成等待阶段展示轮播式、有趣、不重复感强的提示文案。
4. 图片区域等待状态使用方形骨架屏，避免纯白或单调圆点。
5. 保持现有文字模式 fallback，不因为图片处理失败阻塞游戏。

### 不在本次范围

1. 不改游戏评分规则。
2. 不改 LLM 猜词、判题 prompt。
3. 不改 SSE 事件协议，除非后续实现中发现必须扩展。
4. 不新增全局状态库。
5. 不做图片持久化存储。

---

## 推荐方案总览

推荐采用“后端确定性模糊 + 前端轮播等待体验 + 图片骨架屏增强”的方案。

### 为什么不继续只靠 prompt 模糊

只靠 prompt 的问题：

- 模型不一定严格执行。
- 不同图片模型、代理服务、版本更新后表现不稳定。
- 简单模式的“slightly blurry”尤其容易被模型解释成高质量柔焦图，而不是游戏需要的遮蔽式模糊图。

### 为什么后端做高斯模糊

后端是生成图片的唯一入口。把模糊放后端可以保证：

- 所有客户端拿到的猜题图都已经模糊。
- 难度配置和模糊强度在同一处维护。
- 测试可以直接验证 `generate_image` 的输出链路。
- 前端不需要承担安全边界；玩家无法通过 CSS 或 DOM 拿到未模糊版本。

## 后端优化设计

### 1. 新增依赖

在 `backend/requirements.txt` 增加：

```text
Pillow>=10.0
```

用途：

- 解析图片 base64。
- 执行 `ImageFilter.GaussianBlur`。
- 重新编码为 PNG data URL。

### 2. 配置模糊半径

在 `backend/config.py` 的 `DIFFICULTY_CONFIG` 中新增 `blur_radius`：

```python
"easy": {
    "blur_radius": 6,
}

"medium": {
    "blur_radius": 10,
}

"hard": {
    "blur_radius": 14,
}
```

建议初始值：

| 难度 | blur_radius | 目标体验 |
|---|---:|---|
| easy | 6 | 能看出大轮廓，但细节不可读 |
| medium | 10 | 只能判断形状、色块和局部结构 |
| hard | 14 | 高度抽象，必须结合提示猜测 |

这些值应作为第一版默认值，后续可以基于实际试玩微调。

### 3. 调整图片生成函数签名

当前函数：

```python
def generate_image(visual_desc, blur_prompt, image_strategy):
```

建议改为：

```python
def generate_image(visual_desc, blur_prompt, image_strategy, blur_radius):
```

调用方同步更新：

- `backend/services/game_service.py`
- `backend/app.py`

### 4. 图片后处理职责划分

建议在 `backend/services/image_service.py` 内部增加小函数：

```python
def _blur_data_image(image_data_url, radius):
    """把 data:image/*;base64,... 转成模糊后的 data:image/png;base64,...。"""
```

职责：

1. 只接受 `data:image/...;base64,...`。
2. 解码图片内容。
3. 使用 Pillow 打开图片。
4. 转成 RGB/RGBA 兼容格式。
5. 应用高斯模糊。
6. 输出 PNG data URL。

如果图片是远程 URL，有两个可选策略：

#### 推荐策略 A：先不处理远程 URL，返回 `None`

优点：

- 安全简单。
- 避免后端再次请求外部 URL 带来的超时、SSRF 风险和复杂错误处理。
- 当前测试和常见 OpenAI 图片响应已支持 `b64_json`。

缺点：

- 如果代理只返回 URL，猜题图会进入文字模式 fallback。

#### 可选策略 B：下载远程 URL 后模糊

优点：

- 兼容更多图片 API 返回格式。

缺点：

- 需要处理请求超时、Content-Type、文件大小上限、SSRF 风险。
- 超出本次最小可靠优化范围。

建议第一版使用策略 A，除非确认当前图片代理只返回 URL。

### 5. 防止未模糊图泄漏

`generate_image` 的逻辑应是：

```text
请求图片 API -> 规范化图片响应 -> 执行高斯模糊 -> 返回模糊 data URL
```

如果任一步失败：

```text
返回 None -> game_service._build_question 进入 image_mode = text
```

不要在模糊失败时返回原图。原因是原图可能直接暴露答案，破坏游戏公平性。

### 6. 清晰图保持不变

`generate_clear_image` 用于答对后的 reveal 阶段，应继续返回清晰图，不做高斯模糊。

当前职责边界合理：

- `generate_image`：猜题阶段，必须模糊。
- `generate_clear_image`：揭晓阶段，必须清晰。

## 前端优化设计

### 1. 等待提示轮播

建议新增纯函数文件：

```text
frontend/src/pages/loadingPrompts.js
```

内容示例：

```javascript
export const IMAGE_WAITING_PROMPTS = [
  'AI 正在把答案藏进雾里...',
  '图像生成中，先从分类里抓线索。',
  '正在给画面加一层神秘滤镜。',
  '关键词已就位，模糊图马上登场。',
  '别眨眼，轮廓正在慢慢浮现。',
]

export function getWaitingPrompt(index) {
  return IMAGE_WAITING_PROMPTS[index % IMAGE_WAITING_PROMPTS.length]
}
```

这样做的好处：

- 文案集中维护。
- 方便测试。
- 不把轮播文案散落在 hook 中。

### 2. 轮播状态放在 `useGame`

在 `frontend/src/hooks/useGame.js` 中新增状态：

```javascript
const [waitingPromptIndex, setWaitingPromptIndex] = useState(0)
```

在 partial 阶段启动 interval：

```javascript
useEffect(() => {
  if (phase !== 'partial') return undefined

  const timer = window.setInterval(() => {
    setWaitingPromptIndex(prev => prev + 1)
  }, 2400)

  return () => window.clearInterval(timer)
}, [phase])
```

返回值中暴露：

```javascript
waitingPrompt: getWaitingPrompt(waitingPromptIndex)
```

这样 `Game.jsx` 和 `questionPresentation.js` 不需要知道轮播细节。

### 3. 替换固定 fallback 文案

当前硬编码：

```javascript
setFallbackHint('先根据分类判断，图片马上补齐。')
```

建议替换为更语义化的状态：

```javascript
setFallbackHint('')
```

等待文案由 `waitingPrompt` 提供，真正的 fallbackHint 只用于图片失败后的文字模式。

原因：

- partial 阶段不是“图片失败”，只是“图片仍在生成”。
- fallbackHint 应保留给 text mode 使用。
- 轮播等待文案和答题线索应分开，避免职责混乱。

### 4. `questionPresentation` 支持 partial 等待文案

当前函数签名：

```javascript
getQuestionPresentation({ image, category, imageMode, fallbackHint, imageLoadFailed })
```

建议增加：

```javascript
waitingPrompt
```

partial 返回：

```javascript
return {
  mode,
  src: null,
  badge: category || '题目已就绪',
  primaryHint: waitingPrompt || '关键词已就绪，图片还在生成中。',
  notice: '关键词已就绪，图片还在生成中',
}
```

text mode 仍使用 `fallbackHint`。

### 5. 图片骨架屏组件化

建议在 `Game.jsx` 内先抽一个小组件，避免新增过多文件：

```jsx
function ImageSkeleton({ prompt }) {
  return (
    <div className="gm-image-skeleton" aria-live="polite">
      <div className="gm-image-skeleton-grid" aria-hidden="true">
        <span className="gm-skeleton-block gm-skeleton-hero" />
        <span className="gm-skeleton-block" />
        <span className="gm-skeleton-block" />
        <span className="gm-skeleton-block gm-skeleton-wide" />
      </div>
      <p className="gm-image-skeleton-text">{prompt}</p>
    </div>
  )
}
```

适用场景：

- 图片 URL 已有但 `<img>` 还没 load。
- partial 阶段图片尚未返回。

### 6. 方形骨架屏样式

当前 `.gm-image-wrap` 已经是 `aspect-ratio: 1`，可以复用。

建议样式方向：

```css
.gm-image-skeleton {
  position: absolute;
  inset: 0;
  display: grid;
  place-items: center;
  padding: 28px;
  background:
    radial-gradient(circle at 22% 18%, rgba(13, 148, 136, 0.16), transparent 28%),
    linear-gradient(145deg, #f8fafc, #e0f2fe 52%, #f0fdfa);
}

.gm-image-skeleton-grid {
  width: min(72%, 260px);
  aspect-ratio: 1;
  display: grid;
  grid-template-columns: repeat(2, 1fr);
  gap: 12px;
}

.gm-skeleton-block {
  border-radius: 18px;
  background: linear-gradient(90deg, rgba(255,255,255,0.58), rgba(255,255,255,0.9), rgba(255,255,255,0.58));
  background-size: 220% 100%;
  animation: gmSkeletonSweep 1.6s ease-in-out infinite;
}
```

视觉目标：

- 保持方形图片感。
- 有块状结构，暗示“图像正在拼出来”。
- 背景不是纯白，有轻微渐变和空气感。
- 动画轻量，不抢答题注意力。

### 7. partial 阶段也显示骨架屏

当前 partial 阶段因为没有 `presentation.src`，会走 `gm-text-card`。

建议调整 `Game.jsx` 渲染：

```jsx
{presentation.mode === 'partial' ? (
  <ImageSkeleton prompt={presentation.primaryHint} />
) : presentation.src ? (
  ...image rendering...
) : (
  ...text card...
)}
```

这样图片生成中会看到方形骨架屏，而不是文字卡片。

## 分步实施计划

修改文件：

- `backend/requirements.txt`
- `backend/config.py`
- `backend/services/image_service.py`
- `backend/services/game_service.py`

- 图片生成等待期间提示会轮播。

### 阶段 3：方形骨架屏

修改文件：

- 图片等待区域始终保持方形。
- 视觉不再是纯白或单圆点。

- 加载完成后骨架屏自然消失。

## 风险与注意事项

### 1. Pillow 安装风险

`Pillow` 是成熟依赖，但部署环境需要能安装其 wheel。如果服务器环境较老，可能出现编译依赖问题。

缓解方式：

- 使用 `Pillow>=10.0`，优先安装预编译 wheel。
- 在部署前运行 `pip install -r requirements.txt` 验证。

### 2. 模糊半径需要试玩调参

不同图片尺寸下同一个半径的体感不同。当前默认图片尺寸是 `1024x1024`，建议初始使用：

- easy: 6
- medium: 10
- hard: 14

如果后续 `IMAGE_SIZE` 调小，可能需要同步降低半径。

### 3. 不应把原图传给前端再用 CSS 模糊

CSS 模糊虽然实现简单，但不推荐。

原因：

- 玩家可以在浏览器 DevTools 中拿到原始图片。
- 图片加载完成前后可能短暂闪现清晰图。
- CSS 只是展示层效果，不是玩法安全边界。

### 4. 等待提示不能伪装成游戏线索

轮播文案应该是氛围提示，不应提供额外答案线索，否则会影响难度平衡。

推荐文案类型：

- “AI 正在把答案藏进雾里...”
- “关键词已就位，模糊图马上登场。”
- “图像生成中，先从分类里抓线索。”

避免文案类型：

- 暗示答案类别之外的内容。
- 暗示形状、颜色、用途。
- 根据具体题目动态生成额外提示。

### 5. partial 和 text mode 要保持语义分离

- partial：图片还在生成，玩家暂时等待。
- text：图片生成失败，玩家用文字提示继续。

不要让 partial 复用 text fallback 逻辑，否则 UI 和测试会越来越难维护。

## 推荐结论

本次优化的关键不在于修改 prompt，而是把“模糊”从模型自由发挥变成后端确定性处理。

推荐优先级：

1. 先做后端高斯模糊，保证玩法正确性。
2. 再做前端等待提示轮播，提升等待体验。
3. 最后做方形骨架屏，提升视觉完整度。

这样可以让每一步都有独立收益，也便于测试和回滚。
