# AI默契猜词大挑战 - 产品需求文档

## 1. 产品概述

### 1.1 核心定位

轻量级AI互动猜词游戏，结合"文字提示 + 模糊识图"玩法。LLM负责出词与智能判分，AI绘图生成模糊关键词图片，考验玩家联想能力。规则简单易上手，适合好友互动或单人挑战。

### 1.2 目标用户

- 喜欢轻量休闲小游戏的用户
- 朋友间互动娱乐的社交场景
- 对AI能力感兴趣的尝鲜用户

---

## 2. 游戏模式

### 2.1 单人挑战模式

- 玩家独自进行猜词挑战，累计正确次数和积分
- 每局包含一组关键词（默认10个），限时完成
- 结算时显示总得分、正确率、用时等统计

### 2.2 双人对战模式（MVP后迭代）

- 双人同屏轮流猜词，各自计分
- 本地对战优先（共用一台设备，轮流操作），降低技术复杂度
- 每人看到相同的模糊图片，分别作答，对比得分

---

## 3. 核心玩法流程

```
模式/难度选择 → 关键词生成 → 模糊图片生成 → 猜词互动 → 判分反馈 → 解锁清晰图 → 结算
```

### 3.1 模式与难度选择

玩家选择游戏模式和难度等级：

| 难度 | 词库范围 | 图片模糊程度 | 提示次数 | 单题时限 | 基础分值 |
|------|---------|------------|---------|---------|---------|
| 简单 | 常见动物、食物、日用品 | 中度模糊（轮廓可辨） | 3次 | 60秒 | 10分 |
| 中等 | 职业、运动、地标建筑 | 较强模糊（需联想） | 2次 | 45秒 | 20分 |
| 困难 | 抽象概念、成语、复合词汇 | 高度模糊（极少线索） | 1次 | 30秒 | 40分 |

### 3.2 关键词生成

- LLM根据难度等级，随机生成关键词
- 同时生成关键词的**核心视觉特征描述**（用于传给绘图API）
- 同一局游戏内不重复出词，历史已出词通过上下文传递给LLM避免

**LLM Prompt设计要点：**
```
角色：你是一个猜词游戏的出题官。
任务：根据难度"{difficulty}"，生成1个关键词和对应的视觉描述。
要求：
1. 关键词应有明确的视觉表现，适合用图片表达
2. 返回JSON格式：{"keyword": "xxx", "visual_desc": "xxx", "category": "xxx", "hint1": "xxx", "hint2": "xxx", "hint3": "xxx"}
3. visual_desc 用于生成图片，应描述关键词的核心视觉特征，但不包含关键词本身
4. hint1/2/3 为由浅入深的文字提示，不能直接包含答案
5. 避免与以下已出词重复：{used_words}
```

### 3.3 模糊图片生成

调用GPT-image-2生成关键词对应的图片，通过Prompt控制模糊效果：

**不同难度的图片生成策略：**

| 难度 | Prompt策略 | 效果 |
|------|-----------|------|
| 简单 | 基于visual_desc生成完整图片，添加"slightly blurry, soft focus"描述 | 轮廓清晰，细节模糊 |
| 中等 | 基于visual_desc生成局部特写或非典型角度，添加"blurry, out of focus, abstract"描述 | 需要联想才能识别 |
| 困难 | 仅提取visual_desc中的抽象元素，添加"very blurry, heavily abstracted, minimal detail"描述 | 极少视觉线索 |

**图片参数：**
- 模型：`gpt-image-2-all`
- 尺寸：`1024x1024`
- 数量：1张

### 3.4 猜词互动

1. 展示模糊图片，倒计时开始
2. 玩家输入答案
3. LLM实时判定答案，返回判定结果
4. 可消耗提示次数获取文字提示（每次提示扣减当题30%的基础分值）
5. 答对或时间耗尽后进入下一题

### 3.5 智能判分系统

LLM判定玩家答案与关键词的关系，返回结构化结果：

**判分Prompt设计要点：**
```
角色：你是猜词游戏的裁判。
任务：判断玩家答案与正确关键词的匹配程度。
正确关键词："{keyword}"
玩家答案："{answer}"
判定规则：
1. 完全正确（同义词也算）：答案与关键词含义完全一致 → 100%得分
2. 非常接近：答案与关键词高度相关，属于近义词或上下位概念 → 60%-80%得分
3. 有点沾边：答案与关键词有一定关联但差距较大 → 20%-40%得分
4. 完全错误：答案与关键词无关 → 0分
返回JSON格式：{"match": "exact|close|related|wrong", "score_ratio": 0.0-1.0, "feedback": "给玩家的反馈语"}

示例：
- 关键词"猫"，答案"小猫" → exact, 1.0
- 关键词"猫"，答案"猫咪" → exact, 1.0
- 关键词"猫"，答案"布偶" → close, 0.7（猫的品种）
- 关键词"猫"，答案"宠物" → related, 0.3（上位概念）
- 关键词"猫"，答案"汽车" → wrong, 0.0
```

**得分计算公式：**
```
实际得分 = 基础分值 x score_ratio x 时间奖励系数 x 提示惩罚系数

时间奖励系数 = 剩余时间占比（0.5 ~ 1.0，答得越快分越高）
提示惩罚系数 = 1.0 - (已用提示次数 x 0.3)
```

### 3.6 解锁清晰图与知识卡片

猜对关键词（score_ratio >= 0.6）后：

1. 调用GPT-image-2重新生成**高清完整图片**（去掉模糊相关的Prompt）
2. LLM同步生成关键词的**趣味知识卡片**（1-2句有趣的冷知识或科普）
3. 展示清晰图片 + 知识卡片，停留3-5秒后进入下一题

### 3.7 结算页面

一局结束后展示：
- 总得分 / 最高可能得分
- 正确率（exact + close占比）
- 平均用时
- 最佳答题（用时最短或得分最高的一题回顾）
- 分享按钮（可生成结果卡片图片）

## 4. 提示系统

当玩家卡住时，可以使用提示来获取额外线索：

| 提示类型 | 内容 | 代价 |
|---------|------|------|
| 文字提示 | LLM生成的递进式文字提示（hint1 → hint2 → hint3） | 扣减当题30%基础分 |
| 图片清晰化 | 生成一张稍微清晰一点的图片（调整模糊参数） | 扣减当题20%基础分 |

提示按钮在倒计时过半后高亮提醒，引导卡住的玩家使用。

## 5. 积分与奖品系统

### 5.1 积分体系

| 项目 | 说明 |
|------|------|
| 积分来源 | 每题实际得分累加 |
| 连续答对加成 | 连续答对3题+10%，5题+20%，10题+50% |
| 积分存储 | 本地localStorage，后续可扩展至服务端 |

### 5.2 奖品场（可选功能，MVP后迭代）

- 休闲场：纯娱乐，无奖品
- 奖品场：达到一定积分门槛可兑换奖品
- 奖品类型和兑换规则由运营配置，前端读取配置文件渲染

## 6. 技术方案

### 6.1 技术栈

| 层级 | 选型 | 说明 |
|------|------|------|
| 前端 | React + Vite | 组件化开发，状态管理清晰，HMR快速迭代 |
| 后端 | Flask (Python) | API代理层，保护密钥，处理限流，管理游戏会话 |
| 前后端通信 | REST API + JSON | Flask提供接口，React通过fetch/axios调用 |
| LLM API | Gemini（OpenAI兼容格式） | 出词、判分、生成提示和知识卡片 |
| 图片生成 | GPT-image-2 | 生成模糊图和高清图 |
| 图片理解 | Gemini 图片理解API | 备用：可用于验证生成图片是否泄露关键词 |
| 图片编辑 | GPT-image-1.5（编辑接口） | 备用：对已有图片做模糊化/清晰化处理 |
| 数据存储 | SQLite + localStorage | SQLite存储游戏记录与积分，localStorage缓存前端状态 |

### 6.3 Flask后端API设计

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/game/start` | POST | 开始新游戏，传入难度，返回session_id |
| `/api/game/next` | POST | 获取下一题（关键词 + 模糊图片），支持预加载 |
| `/api/game/guess` | POST | 提交答案，返回判分结果 |
| `/api/game/hint` | POST | 获取提示（返回对应级别的hint） |
| `/api/game/reveal` | POST | 猜对后获取清晰图 + 知识卡片 |
| `/api/game/result` | GET | 获取本局结算数据 |
| `/api/game/history` | GET | 获取历史战绩 |

### 6.4 API调用流程

```
┌─────────────┐
│  玩家选择难度  │
└──────┬──────┘
       ▼
┌─────────────────────────────┐
│  LLM: 生成关键词 + 视觉描述    │  ← Gemini Chat API
│  + 提示语 + 分类              │
└──────┬──────────────────────┘
       ▼
┌─────────────────────────────┐
│  GPT-image-2: 生成模糊图片     │  ← /v1/images/generations
│  (prompt = 视觉描述 + 模糊指令) │
└──────┬──────────────────────┘
       ▼
┌─────────────────────────────┐
│  玩家看图猜词，输入答案         │
└──────┬──────────────────────┘
       ▼
┌─────────────────────────────┐
│  LLM: 判定答案匹配度           │  ← Gemini Chat API
│  返回 match + score_ratio     │
└──────┬──────────────────────┘
       ▼
┌──────┴──────────────────────┐
│  猜对(>=0.6)?               │
│  ├─ 是 → GPT-image-2生成清晰图│

│  └─ 否 → 显示正确答案         │
└─────────────────────────────┘
```

### 6.5 API参数速查

**Gemini Chat（LLM判分/出词）：**
- 端点：`/v1/chat/completions`
- 模型：`gemini-2.5-flash`（速度优先，控制成本）
- temperature：0.8（出词时增加随机性）/ 0.1（判分时保证稳定性）
- response_format：`{ "type": "json_object" }`（强制JSON输出）

**GPT-image-2（图片生成）：**
- 端点：`/v1/images/generations`
- 模型：`gpt-image-2-all`
- 尺寸：`1024x1024`
- 数量：1

**Gemini 图片理解（备用校验）：**
- 端点：`/v1beta/models/gemini-2.0-flash:generateContent`
- 用途：校验生成的模糊图片是否直接暴露了关键词

### 6.6 性能与等待体验

| 环节 | 预计耗时 | 等待体验方案 |
|------|---------|------------|
| LLM出词 | 1-3秒 | 展示"AI正在出题..."动画 |
| 图片生成 | 5-15秒 | 先展示文字提示预热，图片就绪后淡入 |
| LLM判分 | 1-2秒 | 展示"AI裁判分析中..."动画 |
| 清晰图生成 | 5-15秒 | 先展示判分结果和知识卡片，图片就绪后替换 |

**优化策略：**
- 当前题作答时，**预加载下一题的关键词和模糊图片**，减少等待
- 判分和清晰图生成可以并行请求
- 首题加载可展示游戏规则/教程作为缓冲

### 6.7 成本预估

| API调用 | 每题次数 | 10题一局 |
|---------|---------|---------|
| LLM出词 | 1次 | 10次 |
| LLM判分 | 1-3次（允许多次作答） | 10-30次 |
| LLM知识卡片 | 0-1次 | ~7次（假设70%答对） |
| 图片生成（模糊） | 1次 | 10次 |
| 图片生成（清晰） | 0-1次 | ~7次 |
| **合计** | | **LLM ~47次 + 图片 ~17次** |

> 需根据实际API定价计算单局成本，建议为每局设置API调用上限做兜底。

## 8. 防作弊与边界处理

| 场景 | 处理方案 |
|------|---------|
| 图片直接暴露关键词 | 生成图片时Prompt明确要求不含文字；可选用Gemini图片理解校验 |
| 玩家恶意快速提交 | 每题最多允许3次作答，超过直接判定失败 |
| LLM判分不一致 | 设定温度为0.1降低随机性；关键判定加入few-shot示例 |
| API超时/失败 | 单次请求设置15秒超时，失败后允许重试1次，仍失败则跳过本题不扣分 |
| 图片生成耗时过长 | 超过20秒未返回，使用纯文字提示模式降级 |
| 关键词不适合图片表达 | LLM出词时要求返回视觉表现力评分，过低则重新生成 |

## 10. 关键风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 图片生成质量不稳定 | 有时太清晰暴露答案，有时太模糊无法辨认 | 多次测试Prompt模板，建立不同难度的Prompt库；极端情况降级为纯文字模式 |
| LLM判分存在歧义 | 边界答案判定不准 | 使用JSON结构化输出 + few-shot示例约束；温度设低 |
| API成本较高 | 单局调用多次API | 预加载复用；考虑缓存高频关键词的图片；限制每日免费局数 |
| 图片生成延迟 | 影响游戏节奏 | 预加载机制 + 等待动画 + 降级方案 |
